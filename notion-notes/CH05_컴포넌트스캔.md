# CH05. 컴포넌트 스캔 (@ComponentScan, @Component)

---

## 이 장의 핵심 한 줄 요약

> **@Bean으로 하나하나 등록하지 않고, @Component를 붙이면 스프링이 자동으로 찾아서 Bean으로 등록해준다.**

---

## 하고 싶은 말이 뭔데?

### CH03~04 방식 (수동 등록)
```java
@Configuration
public class AppContext {
    @Bean public MemberDao memberDao() { return new MemberDao(); }
    @Bean public MemberRegisterService memberRegisterService() { ... }
    @Bean public ChangePasswordService changePasswordService() { ... }
    @Bean public MemberListPrinter memberListPrinter() { ... }
    @Bean public MemberInfoPrinter memberInfoPrinter() { ... }
    // ... 5개, 10개, 20개... 계속 늘어남
}
```

**문제: Bean이 많아지면 설정 클래스가 엄청 길어진다!**

### CH05 방식 (자동 등록)
```java
@Component                                  // 1) 클래스에 @Component 붙이기
public class MemberDao { ... }

@Component
public class MemberRegisterService {
    @Autowired private MemberDao memberDao; // 2) 의존은 @Autowired로 자동 주입
}

@Configuration
@ComponentScan(basePackages = {"org.example.chapter05"})  // 3) 이 패키지를 스캔해!
public class AppContext {
    // MemberDao, MemberRegisterService 등은 @Bean 안 써도 자동 등록됨!
    // 자동 스캔이 안 되는 것만 @Bean으로 수동 등록
    @Bean public VersionPrinter versionPrinter() { ... }
}
```

---

## 반드시 알아야 할 핵심 개념

### 1. @Component - "나를 Bean으로 등록해줘"

클래스 위에 붙이면 스프링이 자동으로 Bean으로 등록한다.

```java
@Component      // Bean 이름 = "memberDao" (클래스명 첫 글자 소문자)
public class MemberDao { ... }

@Component("memberListPrinter")     // Bean 이름을 직접 지정할 수도 있음
public class MemberListPrinter { ... }
```

**Bean 이름 규칙:**

| 상황 | Bean 이름 |
|---|---|
| `@Component` (이름 생략) | 클래스명 첫 글자를 소문자로 -> `MemberDao` -> `"memberDao"` |
| `@Component("myName")` (이름 지정) | 지정한 이름 -> `"myName"` |

### 2. @ComponentScan - "이 패키지에서 @Component를 찾아라"

```java
@Configuration
@ComponentScan(basePackages = {"org.example.chapter05"})
public class AppContext { ... }
```

- `basePackages`: 스캔할 패키지 경로 (하위 패키지 포함)
- 해당 패키지 안에서 `@Component`가 붙은 클래스를 전부 찾아 Bean으로 등록

### 3. excludeFilters - 특정 클래스를 스캔에서 제외

스캔 범위에서 특정 클래스를 빼고 싶을 때:

#### (1) ASSIGNABLE_TYPE - 특정 클래스/인터페이스 제외

```java
@ComponentScan(
    basePackages = "org.example.chapter05",
    excludeFilters = @ComponentScan.Filter(
        type = FilterType.ASSIGNABLE_TYPE,
        classes = {MemberDao.class}           // MemberDao 제외
    )
)
```

#### (2) REGEX - 정규식으로 제외

```java
@ComponentScan(
    basePackages = "org.example.chapter05",
    excludeFilters = @ComponentScan.Filter(
        type = FilterType.REGEX,
        pattern = "org\\.example\\.chapter05\\..*Dao"    // ~Dao로 끝나는 클래스 제외
    )
)
```

#### (3) ASPECTJ - AspectJ 패턴으로 제외

```java
@ComponentScan(
    basePackages = "org.example.chapter05",
    excludeFilters = @ComponentScan.Filter(
        type = FilterType.ASPECTJ,
        pattern = "org.example.chapter05.*Dao"    // ~Dao로 끝나는 클래스 제외
    )
)
```

#### 필터 여러 개 조합

```java
excludeFilters = {
    @ComponentScan.Filter(type = FilterType.ASSIGNABLE_TYPE, classes = AppContext.class),
    @ComponentScan.Filter(type = FilterType.REGEX, pattern = "org\\.example\\..*Dao")
}
```

### 4. Bean 이름 충돌 (중요!)

**같은 이름의 Bean이 2개 이상이면 에러!**

```java
// 패키지 A
@Component
public class MemberRegisterService { }

// 패키지 B (다른 패키지, 같은 클래스명)
@Component
public class MemberRegisterService { }
```

둘 다 스캔하면 Bean 이름이 `"memberRegisterService"`로 같아서 -> `BeanDefinitionStoreException` 에러!

**해결 방법:**
```java
@Component("chapter03MemberRegisterService")    // 이름을 다르게 지정
public class MemberRegisterService { }

@Component("chapter05MemberRegisterService")    // 이름을 다르게 지정
public class MemberRegisterService { }
```

### 5. 수동 등록(@Bean) vs 자동 스캔(@Component) 충돌

**같은 이름**으로 `@Bean`과 `@Component` 둘 다 등록하면?

```java
@Component
public class MemberDao { ... }     // Bean 이름 = "memberDao"

// 설정 클래스에서도 같은 이름으로 수동 등록
@Bean
public MemberDao memberDao() {     // Bean 이름 = "memberDao" (같음!)
    return new ManualMemberDao();   // MemberDao 상속
}
```

-> **같은 이름일 때, 수동 등록(@Bean)이 우선!** 자동 스캔된 Bean을 수동 Bean이 대체한다.

**주의:** 이름이 다르면 두 Bean이 **동시에 존재**하고, `@Autowired` 시 같은 타입이 2개이므로 타입 충돌(`NoUniqueBeanDefinitionException`)이 발생할 수 있다. 이때는 `@Qualifier`로 해결.

## CH03 -> CH04 -> CH05 흐름 정리

```
CH03: @Bean으로 전부 수동 등록 + 수동으로 의존 연결
        |
        v
CH04: @Bean으로 수동 등록 + @Autowired로 자동 주입
        |
        v
CH05: @Component로 자동 등록 + @Autowired로 자동 주입
        = 설정 코드가 최소화됨!
```

| 챕터 | Bean 등록 | 의존 주입 |
|---|---|---|
| CH03 | `@Bean` (수동) | `@Bean` 메서드 내에서 직접 (수동) |
| CH04 | `@Bean` (수동) | `@Autowired` (자동) |
| CH05 | `@Component` + `@ComponentScan` (자동) | `@Autowired` (자동) |

---

## Spring Boot와 뭐가 다른가 / 생소한 부분

| Spring Boot에서는 | 이 챕터에서는 |
|---|---|
| `@SpringBootApplication`에 `@ComponentScan` 포함됨 | `@ComponentScan`을 직접 명시 |
| `basePackages` 안 써도 메인 클래스 패키지부터 자동 스캔 | `basePackages`를 보통 명시적으로 지정 (생략 시 설정 클래스 패키지 기준 스캔) |
| `excludeFilters` 거의 안 씀 | `excludeFilters`의 다양한 타입을 학습 |
| `@Service`, `@Repository` 등 세부 어노테이션 사용 | `@Component`만 사용 |

**참고:** `@Service`, `@Repository`, `@Controller`는 모두 `@Component`를 포함한다. 기능은 같고 역할 구분용이다.

## 시험 출제 포인트

### 객관식/단답형

1. **@Component의 역할은?** -> 해당 클래스를 스프링 Bean으로 자동 등록
2. **@ComponentScan의 역할은?** -> 지정된 패키지에서 @Component가 붙은 클래스를 찾아 Bean으로 등록
3. **@Component만 붙이고 이름을 지정하지 않으면 Bean 이름은?** -> 클래스명의 첫 글자를 소문자로 바꾼 것
4. **같은 Bean 이름이 충돌하면?** -> BeanDefinitionStoreException 발생
5. **같은 이름의 @Bean(수동)과 @Component(자동)이 충돌하면?** -> 같은 이름일 때 @Bean(수동)이 우선하여 대체. 이름이 다르면 둘 다 존재하며 타입 충돌 가능

### 서술형

1. **@ComponentScan의 excludeFilters에서 사용 가능한 FilterType 3가지를 설명하시오**
   - `ASSIGNABLE_TYPE`: 지정한 클래스/인터페이스 및 하위 타입을 제외
   - `REGEX`: 정규표현식 패턴과 일치하는 클래스를 제외
   - `ASPECTJ`: AspectJ 표현식 패턴과 일치하는 클래스를 제외

2. **CH03(수동 등록)에서 CH05(컴포넌트 스캔)으로 발전하는 과정을 설명하시오**
   - CH03: 모든 Bean을 @Bean 메서드로 수동 등록, 의존 관계도 직접 코드로 연결
   - CH04: Bean은 수동 등록하되 @Autowired로 의존 자동 주입 도입
   - CH05: @Component + @ComponentScan으로 Bean 등록까지 자동화, 설정 코드 최소화

---

## 이해도 체크리스트

- [ ] @Component와 @ComponentScan의 관계를 설명할 수 있다
- [ ] @Component의 Bean 이름 규칙 (기본값 vs 직접 지정)을 안다
- [ ] excludeFilters의 3가지 FilterType을 구분할 수 있다
- [ ] Bean 이름 충돌이 왜 생기고 어떻게 해결하는지 안다
- [ ] 같은 이름의 @Bean(수동)이 @Component(자동)보다 우선한다는 것, 이름이 다르면 둘 다 존재한다는 것을 안다
- [ ] CH03 -> CH04 -> CH05 발전 과정을 한 문장씩으로 요약할 수 있다

## 검토 보강 포인트

- `@Component`를 붙였다고 바로 Bean이 되는 게 아니라, **그 클래스를 스캔하는 `@ComponentScan` 또는 다른 등록 경로가 있어야** 실제로 Bean으로 등록된다.
- `@ComponentScan(basePackages = "...")`는 명시적으로 범위를 주는 방식이고, **인자를 생략하면 설정 클래스가 속한 패키지와 그 하위 패키지**를 기본으로 스캔한다. 즉, `basePackages`는 "반드시" 써야 하는 것은 아니다.
- `@Service`, `@Repository`, `@Controller`는 모두 `@Component`의 세부 형태다. 시험에서는 **역할 구분용 stereotype annotation** 이라는 점까지 같이 기억하면 좋다.
- **Bean 이름 충돌**과 **타입 충돌**은 다르다. 이름이 같으면 등록 단계에서 충돌하고, 이름은 달라도 타입이 같으면 주입 단계에서 `@Autowired`가 모호해질 수 있다.
- 수동 `@Bean`이 자동 스캔 Bean보다 우선하는 상황은 **같은 이름으로 등록될 때**로 이해하는 게 정확하다. 이름이 다르면 두 Bean이 같이 존재하고, 그 경우에는 오히려 자동 주입 시 타입 충돌을 신경 써야 한다.
- 이 장의 실제 예제에는 `excludeFilters`, 이름 충돌 해결, 수동 Bean 우선 예제까지 들어 있으므로, 본문 개념만 외우지 말고 **예외 케이스를 묻는 문제**도 대비하는 것이 좋다.
