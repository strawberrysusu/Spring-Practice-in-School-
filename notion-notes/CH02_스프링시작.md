# CH02. 스프링 시작하기

---

## 이 장의 핵심 한 줄 요약

> **스프링은 객체(Bean)를 직접 new로 만들지 않고, 설정 클래스(@Configuration)에서 만들어서 컨테이너(ApplicationContext)가 관리하게 한다.**

---

## 하고 싶은 말이 뭔데?

지금까지는 이렇게 했을 거다:

```java
Greeter greeter = new Greeter();        // 내가 직접 만듦
greeter.setFormat("%s, 안녕하세요!");     // 내가 직접 설정
```

스프링에서는 이렇게 한다:

```java
// 1) 설정 클래스에서 "이 객체 이렇게 만들어" 라고 정의
@Configuration
public class AppContext {
    @Bean
    public Greeter greeter() {
        Greeter g = new Greeter();
        g.setFormat("%s, 안녕하세요!");
        return g;
    }
}

// 2) 스프링 컨테이너한테 "greeter 줘" 라고 요청
AnnotationConfigApplicationContext ctx =
    new AnnotationConfigApplicationContext(AppContext.class);
Greeter greeter = ctx.getBean("greeter", Greeter.class);
```

**"왜 굳이?"** -> 객체 생성/관리를 스프링한테 맡기면, 나중에 객체 간 의존 관계가 복잡해져도 스프링이 알아서 조립해준다. CH03부터 그 이유가 드러난다.

---

## 반드시 알아야 할 핵심 개념

### 1. 스프링 컨테이너 (ApplicationContext)

| 개념 | 설명 |
|------|------|
| **스프링 컨테이너** | Bean(객체)을 생성하고, 보관하고, 꺼내주는 "객체 보관소" |
| **AnnotationConfigApplicationContext** | 자바 설정 클래스(@Configuration)를 읽어서 컨테이너를 만드는 구현체 |
| **getBean()** | 컨테이너에서 특정 Bean을 꺼내는 메서드 |

```java
// 컨테이너 생성 (AppContext.class 설정을 읽음)
AnnotationConfigApplicationContext ctx =
    new AnnotationConfigApplicationContext(AppContext.class);

// Bean 꺼내기 ("이름", 타입.class)
Greeter greeter = ctx.getBean("greeter", Greeter.class);
```

### 2. @Configuration

- 클래스 위에 붙이면 **"이 클래스는 스프링 설정 파일이야"** 라는 뜻
- XML 설정 파일 대신 자바 코드로 설정하는 방식 = **JavaConfig**
- 이 클래스 안에 @Bean 메서드를 넣어서 빈을 정의

```java
@Configuration   // "나는 설정 클래스야"
public class AppContext {
    // ... @Bean 메서드들
}
```

### 3. @Bean

- 메서드 위에 붙이면 **"이 메서드가 리턴하는 객체를 Bean으로 등록해"** 라는 뜻
- **메서드 이름 = Bean 이름** (중요!)

```java
@Bean
public Greeter greeter() {   // Bean 이름 = "greeter"
    Greeter g = new Greeter();
    g.setFormat("%s, 안녕하세요!");
    return g;
}
```

### 4. 싱글톤 (Singleton)

스프링 컨테이너는 기본적으로 **Bean을 딱 1개만** 만든다.

```java
Greeter g1 = ctx.getBean("greeter", Greeter.class);
Greeter g2 = ctx.getBean("greeter", Greeter.class);
System.out.println(g1 == g2);  // true!!! 같은 객체!
```

- `getBean()`을 몇 번 호출해도 **같은 객체**를 돌려줌
- 이게 스프링의 **기본 스코프 = 싱글톤**

## Spring Boot와 뭐가 다른가 / 생소한 부분

| Spring Boot에서는 | 이 챕터(순수 Spring)에서는 |
|---|---|
| `@SpringBootApplication` 하나면 끝 | `@Configuration` 클래스를 직접 만들어야 함 |
| `application.properties`로 설정 | 자바 코드로 직접 설정 |
| 내장 톰캣이 자동 실행 | 웹 서버 없음, `main()`에서 직접 실행 |
| Bean 자동 스캔 (`@Component`) | `@Bean` 메서드로 수동 등록 |
| `spring-boot-starter` 의존성 | `spring-context` 하나만 사용 |

**핵심 차이:** Spring Boot는 많은 것을 자동으로 해주지만, 여기서는 모든 것을 수동으로 해서 **스프링이 내부적으로 뭘 하는지** 이해하는 게 목적이다.

## 시험 출제 포인트

### 객관식/단답형으로 나올 수 있는 것들

1. **스프링 컨테이너의 역할은?** -> Bean의 생성, 관리, 제공 (IoC 컨테이너)
2. **@Configuration의 역할은?** -> 해당 클래스가 스프링 설정 정보를 담고 있음을 표시
3. **@Bean의 역할은?** -> 메서드의 리턴 객체를 스프링 컨테이너에 Bean으로 등록
4. **@Bean 메서드의 이름과 Bean 이름의 관계는?** -> 메서드 이름이 곧 Bean 이름
5. **같은 Bean을 두 번 getBean()하면?** -> 같은 객체 (싱글톤)
6. **AnnotationConfigApplicationContext의 생성자 인자는?** -> @Configuration이 붙은 설정 클래스

### 서술형으로 나올 수 있는 것들

1. **스프링 컨테이너를 사용하는 이유를 설명하시오**
   - 객체 생성과 의존 관계 설정을 프레임워크에 위임
   - 코드 변경 없이 설정만 바꾸면 다른 객체로 교체 가능
   - 싱글톤 관리를 자동으로 해줌

2. **JavaConfig 방식과 XML 방식의 차이를 설명하시오**
   - JavaConfig: 자바 코드로 설정, 컴파일 시점에 오류 검출, IDE 자동완성 지원
   - XML: XML 파일로 설정, 런타임에 오류 검출, 코드 수정 없이 설정 변경 가능

## 검토 보강 포인트

- **Bean은 클래스 자체가 아니라, 스프링 컨테이너에 등록되어 관리되는 객체**다. `Greeter` 클래스가 있다고 자동으로 Bean이 되는 게 아니라 `@Bean`으로 등록했기 때문에 Bean이 된다.
- **싱글톤은 "컨테이너 1개 기준"**이다. 같은 `ApplicationContext`에서 `getBean()`을 여러 번 하면 같은 객체를 받지만, 컨테이너를 새로 만들면 새로운 Bean 인스턴스가 생긴다.
- `getBean()`은 이름+타입으로 가져오는 방식 외에 **타입만으로 가져오는 방식**도 있다. 다만 같은 타입 Bean이 여러 개면 나중에 충돌이 날 수 있다.
- 이 장에서는 "컨테이너가 객체를 만들고 관리한다"는 감을 잡는 것이 핵심이다. **의존 관계를 어떻게 연결하는지(DI)** 는 다음 장인 CH03의 중심 주제다.
- 예제에서는 생략되어 있지만, 단독 실행 예제에서는 `AnnotationConfigApplicationContext`를 다 썼으면 **닫아주는 습관**도 알아두면 좋다.
