# 전체 흐름 총정리 (CH02 ~ CH07)

---

## 한눈에 보는 챕터 간 연결 관계

```
CH02 스프링 시작
  "객체를 스프링 컨테이너가 관리한다"
  핵심: @Configuration, @Bean, getBean(), 싱글톤
       |
       v
CH03 스프링 DI
  "객체 간 연결(의존)도 스프링이 해준다"
  핵심: 생성자 주입, setter 주입, @Import, 설정 분리
       |
       v
CH04 자동 주입
  "의존 연결을 자동으로 해준다"
  핵심: @Autowired, @Qualifier, required = false
       |
       v
CH05 컴포넌트 스캔
  "Bean 등록도 자동으로 해준다"
  핵심: @Component, @ComponentScan, excludeFilters
       |
       v
CH06 AOP
  "공통 기능도 자동으로 끼워넣어 준다"
  핵심: @Aspect, @Around, execution, Proxy, @Order
       |
       v
CH07 DB 연동
  "DB 작업도 스프링이 간결하게 해준다"
  핵심: DataSource, JdbcTemplate, RowMapper, @Transactional
```

---

## 각 챕터 핵심 키워드 표

| 챕터 | 핵심 질문 | 핵심 어노테이션/클래스 | 기억할 것 |
|------|----------|---------------------|----------|
| CH02 | Bean을 어떻게 만들고 꺼내나? | `@Configuration`, `@Bean` | `AnnotationConfigApplicationContext`, `getBean()`, **싱글톤** |
| CH03 | 객체 간 의존을 어떻게 연결하나? | `@Bean`, `@Import` | 생성자 주입, setter 주입, **Assembler vs 스프링 컨테이너** |
| CH04 | 의존 주입을 자동으로 할 수 없나? | `@Autowired`, `@Qualifier` | 타입 기반 자동 주입, 같은 타입 2개면 @Qualifier, **required=false** |
| CH05 | Bean 등록도 자동으로 할 수 없나? | `@Component`, `@ComponentScan` | 자동 스캔, Bean 이름 규칙, **excludeFilters**, 이름 충돌 |
| CH06 | 공통 기능을 분리할 수 없나? | `@Aspect`, `@Around`, `@EnableAspectJAutoProxy` | **Proxy 원리**, execution 표현식, `proceed()`, **@Order** |
| CH07 | DB 작업을 간결하게 할 수 없나? | `JdbcTemplate`, `@Transactional`, `@EnableTransactionManagement` | **DataSource + 커넥션 풀(Tomcat JDBC)**, RowMapper, KeyHolder, **트랜잭션 롤백 조건** |

---

## 자동화 발전 과정

| 단계 | Bean 등록 | 의존 주입 | 공통 기능 | DB 연동 |
|------|----------|----------|----------|---------|
| CH02~03 | 수동 (@Bean) | 수동 (코드로 직접) | 수동 (코드 복붙) | - |
| CH04 | 수동 (@Bean) | **자동** (@Autowired) | 수동 | - |
| CH05 | **자동** (@Component) | **자동** (@Autowired) | 수동 | - |
| CH06 | 수동 (@Bean) | 자동 (@Autowired) | **자동** (@Aspect + Proxy) | - |
| CH07 | 혼합 (@Bean 수동 + @ComponentScan 자동) | 자동 (@Autowired) | 자동 | **JdbcTemplate** + **@Transactional** |

> **주의:** CH05에서 "자동 등록 개념"을 배우지만, CH06·CH07의 실제 예제 코드는 @Bean 수동 등록을 계속 사용한다. "챕터가 올라가면 다 자동"이 아니라, **상황에 따라 수동/자동을 섞어 쓴다**는 점을 기억할 것.

---

## CH06 -> CH07 연결 포인트

CH06에서 AOP 프록시를 배웠는데, **@Transactional도 AOP 프록시로 동작한다!**

```
CH06: @Aspect + @Around    -> 프록시가 메서드 감싸서 공통 기능 실행
CH07: @Transactional       -> 프록시가 메서드 감싸서 트랜잭션 관리
                              (시작 -> 커밋 or 롤백)
```

즉, @EnableAspectJAutoProxy(CH06)와 @EnableTransactionManagement(CH07)는 둘 다 **"프록시를 만들어서 메서드를 가로챈다"** 는 같은 원리다.

## 시험 직전 최종 체크 (이것만은 꼭!)

### CH02
- [ ] `@Configuration` + `@Bean` = 스프링 설정
- [ ] `getBean("이름", 타입.class)` = Bean 꺼내기
- [ ] 싱글톤: 같은 Bean을 여러 번 꺼내도 같은 객체

### CH03
- [ ] DI = 외부에서 의존 객체를 넣어주는 것
- [ ] 생성자 주입 vs setter 주입 차이
- [ ] @Import로 설정 클래스 합치기

### CH04
- [ ] @Autowired = 타입 기반 자동 주입
- [ ] 같은 타입 Bean 2개 -> @Qualifier로 구분
- [ ] required = false -> Bean 없어도 에러 안 남
- [ ] 필드/setter/생성자 주입 3가지

### CH05
- [ ] @Component = 자동 Bean 등록
- [ ] @ComponentScan(basePackages) = 스캔 범위 지정
- [ ] Bean 이름 = 클래스명 첫 글자 소문자
- [ ] excludeFilters: ASSIGNABLE_TYPE, REGEX, ASPECTJ
- [ ] 같은 이름일 때 수동(@Bean) > 자동(@Component). 이름 다르면 둘 다 존재

### CH06
- [ ] AOP 5대 용어: Aspect, Advice, Pointcut, JoinPoint, Weaving
- [ ] @Aspect + @Around + execution 표현식
- [ ] proceed() = 원래 메서드 실행
- [ ] @EnableAspectJAutoProxy 필수
- [ ] @Order(1)이 @Order(2)보다 먼저 (바깥쪽)
- [ ] getBean 결과는 프록시 객체

### CH07
- [ ] DataSource = DB 커넥션 제공 인터페이스 (이 예제에서는 Tomcat JDBC 커넥션 풀 사용)
- [ ] JdbcTemplate = JDBC 반복 코드 제거
- [ ] query() = 여러 행, queryForObject() = 단일 값
- [ ] RowMapper = ResultSet -> 객체 변환
- [ ] KeyHolder = INSERT 후 자동 생성 키 확보
- [ ] @Transactional = RuntimeException 시 롤백, 정상 시 커밋
- [ ] @EnableTransactionManagement + DataSourceTransactionManager 필수
- [ ] @Transactional도 AOP 프록시로 동작
